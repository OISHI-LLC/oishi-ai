# Lessons（上限50行。超えたら統合・削除する）

## Deploy
- 本番反映は CI のみ。手動 scp は緊急+ユーザー明示指示時のみ。
- commit に意図しないファイルを含めない。
- `paths-ignore` だけに依存せず、docs-only push を止めたい workflow には changed-files ベースの二重ガードを入れる。

## Verification
- 成果物（画像・HTML等）は本番反映前に必ず目視確認する。
- 「反映済み」は URL/レスポンスで事実確認。ローカル変更だけで完了としない。
- WP スケジュール: `--post_date` と `--post_date_gmt` を両方設定し、`post_status=future` を確認。

## Quality Gate — 納品前に必ず実行（省略禁止）
- **全成果物を出す前に100点チェックを実行し、結果を提示してから納品する。「まあいいか」で出したら失格。**
- 画像（SVG→PNG/webp）は変換後に全数 Read で目視確認。テキストはみ出し・重なり・切れを1枚も見逃さない。
- 記事：ファクトチェック・SEO（タイトル・メタ・見出し・内部リンク・OGP）・HTML構文・文字数・初心者目線・具体例・トーン（上から目線禁止）を全項目確認。
- X投稿：文字数・ファクト・記事との整合性・画像確認を全件実施。
- 明らかな品質問題は聞かずに即修正。壊れた成果物を「直しますか？」と聞くのは失礼。
- rsvg-convert は Unicode 絵文字を描画不可。SVG vector path で代替する。
- PILでのウォーターマーク除去は不向き。1回で失敗したら同じ手法を繰り返さず、ユーザーにCanva等での除去を依頼する。
- 能力の限界を認めて早期に伝える。できないことに何度も挑戦してトークン・時間を浪費しない。

## WordPress / Theme
- favicon は `wp_head` / `login_head` / `admin_head` すべてに設定。
- テーマカラー変更時は CSS 変数だけでなく、ハードコードされた rgba も全て更新。
- ユーザーの参考画像はあくまで方向性。サイトのデザインシステムに合わせる。
- 記事詳細のアイキャッチはテンプレートにラッパークラスを置くだけで終わらせない。`single.php` の画像ラッパーには対応する幅・余白・画像サイズの CSS があることを確認する。

## Code Structure
- 単一ファイルが肥大化したら「最小変更」を言い訳にせず分割する。
- 非自明な拡張が2回以上続いたら責務分離を先に行う。

## Communication
- ユーザーへの応答は常に丁寧語（です/ます）で統一。タメ語を混ぜない。
- ローカル差分の警告等は Git 用語を避け、平易な日本語で伝える。
- コンテンツ制作方針は `PROJECT_RULES.md` を正本にし、記事・X投稿・調査の品質基準をそこへ集約する。
- 記事や投稿で「使うべきではない人」「向いていない人」を強く断定しない。読者の意欲を削ぐ上から目線を避け、注意喚起が必要な場合も「まずはこういう進め方が安全です」のように支援的な言い換えにする。

## Chatbot Patterns
- モデル名検知を広げたら、`会社の名前は？` 等の負例テストも必ず実施。
- Ollama モデル指定はサフィックスまで正確に確認してから作業する。
- チャットボットの文面品質は、実際の自然文プロンプトで確認する。`**強調**` や `|` 区切り表が生表示のままなら未完了。
- PHP から JS/CSS を分離した後は、`node --check` や `php -l` で構文確認し、HTML断片の混入を見逃さない。
- 会社概要まわりで誤回答が出たら、症状ごとの正規表現追加ではなく、構造化データ + 固定応答レイヤーへ切り出してから補助的に文脈判定を足す。
- 最新情報レイヤーは会社情報固定レイヤーと分離し、取得元と取得時刻を必ず返す。
- 日本の天気は Open-Meteo の予報APIだけを使い、地名解決は Nominatim など別 geocoder に分ける。
- 状態付きヒアリングは「1項目ずつ返ってくる」前提にしない。番号付きや複数項目の一括回答を最初から受け付ける。
- チャットボット修正は会話での発見待ちにしない。会社情報 / 相談整理 / 最新情報 / 通常相談の代表質問を回帰セット化して毎回通す。

## Multi-Agent
- handoff.md + PROJECT_RULES.md で引き継ぎ。エージェント固有ファイルは薄く保つ。
- X運用はハッシュタグ禁止、1日3投稿、`1記事 -> 3ツイート` の循環を崩さない。
- 朝運用の自動化は全件キュー消化型にしない。当日必要な候補だけを少数処理する最小フローを優先する。
- X API 投稿は secret の有無だけで安心しない。予約運用前に `create_tweet` の実投稿権限を先に確認し、403 が出たら App permission と token 再発行を最優先で切り分ける。

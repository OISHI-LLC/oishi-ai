# PROJECT_RULES.md

このファイルは、Codex / Claude 共通の運用ルールです。
このプロジェクトでは、`ホームページ運用を止めないこと` と `散らかり防止` を最優先にします。

## 1) まず読む順番

1. `PROJECT_RULES.md`（このファイル）
2. `tasks/handoff.md`（最新の引き継ぎ）
3. 各エージェント専用ファイル（`CLAUDE.md` / `CODEX_ONLY.md`）

## 2) 作業の原則

1. Single Writer ルール:
- 同じ時刻に「書き込み担当」は1エージェントのみ。
- もう1エージェントは調査・レビューのみ実施。

2. 最小変更:
- 目的に必要なファイルだけを触る。
- 影響範囲が広い変更は先に簡単な計画を書く。

3. 事実ベース:
- 「反映済み」は必ず URL/レスポンス/ハッシュで確認する。

## 3) ディレクトリ運用

1. `assets/`:
- 本番で配信されるアセットのみを置く。
- 生成元があるものは `assets/<type>/source.*` を正本にする。

2. `tasks/`:
- 作業ログ、引き継ぎ、バックアップ、補助スクリプトを置く。
- 仕様書の代わりではなく「運用記録」に限定する。

3. `tasks/backups/`:
- ワークストリームごとに baseline 1つを原則とする。
- 中間バックアップは baseline 作成後に整理する。

## 4) デプロイ運用

1. 本番デプロイ経路は **GitHub Actions（`.github/workflows/deploy.yml`）のみ** とする。
2. 通常運用では手動 `scp` / 直接アップロードを禁止する。
3. 例外は「緊急復旧」のみ。ユーザーの明示指示がある場合に限り、手動反映を許可する。
4. デプロイ担当は1回につき1エージェント（Single Writer）とする。
5. デプロイ後、最低限以下を検証する:
- `/`
- `/blog/`
- `/wp-login.php`
- `/favicon.ico`（HTTP 200）
6. `tasks/handoff.md` には「Workflow実行結果（成功/失敗）」を必ず記録する。

## 5) 引き継ぎ運用

1. 作業開始前に `tasks/handoff.md` の最新エントリを読む。
2. 作業完了時、同ファイルに以下を追記する:
- 何を変更したか
- 何を検証したか
- 未解決事項
3. `handoff.md` は直近3件のみ保持する。4件目以降は `handoff-archive.md` へ移動する。
4. `todo.md` は未完了タスクのみ記載する。完了したタスクとReviewは削除する（handoffに記録があるため不要）。
5. `lessons.md` は50行以内を維持する。超えたら定着済みの項目を統合・削除して圧縮する。

## 6) ユーザーの操作を最小化する指示

ユーザーは次の1文だけで良い:

`tasks/handoff.md の最新を読んで、未完了タスクを続けて。デプロイと検証まで実施して。`
